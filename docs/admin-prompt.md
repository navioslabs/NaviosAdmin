# EC管理画面 作成プロンプト

```
あなたはECサイト管理画面の構築に精通したシニアフルスタックエンジニアです。
以下の仕様に基づいて、本番品質のEC管理画面を構築してください。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
## 1. プロジェクト概要
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- **プロジェクト名**: NaviosAdmin
- **目的**: Navios モバイルアプリ連携ECサイトの商品・注文・顧客を一元管理する社内向け管理画面
- **利用者**: 運営スタッフ、マネージャー、システム管理者
- **既存システムとの連携**: Supabase (Auth, Postgres) / Navios モバイルアプリ

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
## 2. 技術スタック（確定済み）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- **フロントエンド**: React 19 (SPA) + TypeScript 5 + Vite 6
- **UIライブラリ**: shadcn/ui + Tailwind CSS 4 + Lucide React
- **状態管理**: React Context（認証） ※必要に応じて追加
- **バックエンド/API**: Hono (Cloudflare Pages Functions)
- **DB**: Supabase (PostgreSQL)
- **認証**: Supabase Auth
- **ホスティング**: Cloudflare Pages（無料枠）
- **ルーティング**: React Router v7
- **その他**: @base-ui/react（shadcn/ui依存）, class-variance-authority, tw-animate-css

上記のスタックに厳密に従ってください。代替ライブラリは提案しないでください。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
## 3. 画面構成と機能要件
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### 3-A. ダッシュボード（トップページ）

以下のKPIカードを表示:
- 今日の売上 / 昨日比 ％表示
- 今月の注文数 / 前月比
- 新規顧客数（今月）
- 在庫切れ商品数（アラート表示）

グラフ:
- 売上推移（日別/週別/月別 切り替え可能な折れ線グラフ）
- カテゴリ別売上構成（円グラフ or 棒グラフ）
- 注文ステータス分布（棒グラフ）

直近アクティビティ:
- 最新注文5件のリスト（ステータスバッジ付き）
- 在庫アラート一覧

### 3-B. 商品管理（Products）

**一覧画面:**
- テーブル表示: 商品画像サムネイル, 商品名, SKU, カテゴリ, 価格, 在庫数, ステータス（公開/下書き/非公開）
- 検索: 商品名・SKUでのインクリメンタルサーチ
- フィルタ: カテゴリ, ステータス, 価格帯, 在庫状況（在庫あり/なし/残少）
- ソート: 各カラムクリックでのソート切り替え
- 一括操作: 複数選択 → ステータス変更 / 一括削除
- ページネーション: 1ページ20件、ページサイズ変更可能
- CSV出力: 現在のフィルタ条件を反映した状態でエクスポート

**作成・編集画面:**
- 基本情報: 商品名, 説明（リッチテキストエディタ）, カテゴリ（階層セレクト）, タグ
- 価格設定: 通常価格, セール価格, セール期間（開始/終了日時）
- 在庫管理: SKU, 在庫数, 在庫追跡ON/OFF
- 画像: 複数画像アップロード（ドラッグ&ドロップ）, 並び替え, メイン画像指定
- SEO: メタタイトル, メタディスクリプション, URLスラッグ
- バリデーション: zodスキーマによるリアルタイムバリデーション
- 下書き保存 / 公開の切り替え

**削除:**
- 論理削除（ソフトデリート）、確認モーダル付き

### 3-C. 注文管理（Orders）

**一覧画面:**
- テーブル表示: 注文番号, 顧客名, 注文日, 合計金額, 決済状況, 配送状況, ステータス
- ステータスバッジ: 色分け表示（新規=青, 処理中=黄, 発送済=緑, キャンセル=赤, 返品=グレー）
- フィルタ: ステータス, 期間指定（DateRangePicker）, 金額帯, 決済方法
- 検索: 注文番号, 顧客名, メールアドレス
- CSV出力

**詳細画面:**
- 注文情報: 注文番号, 日時, ステータス変更ドロップダウン（履歴表示付き）
- 顧客情報: 名前, メール, 電話（顧客詳細へのリンク）
- 商品明細: テーブル形式（商品名, 数量, 単価, 小計）
- 金額サマリー: 小計, 送料, 割引, 税, 合計
- 配送情報: 配送先住所, 追跡番号（入力・更新可能）
- 決済情報: 決済方法, 決済ID, ステータス
- メモ: 社内メモ（スタッフ間共有用テキストエリア）
- アクションボタン: 発送処理, キャンセル, 返金処理

### 3-D. 顧客管理（Customers）

**一覧画面:**
- テーブル表示: 顧客名, メール, 登録日, 累計注文回数, 累計購入金額, ステータス
- 検索: 名前, メール, 電話番号
- フィルタ: 登録期間, 購入金額帯, 注文回数帯
- CSV出力

**詳細画面:**
- プロフィール: 基本情報の表示・編集
- 注文履歴: この顧客の過去注文一覧（注文詳細へリンク）
- 統計: 累計購入金額, 平均注文金額, 最終注文日, LTV
- 配送先住所: 登録済み住所の一覧
- メモ: 社内メモ

### 3-E. 共通レイアウト

**サイドバーナビゲーション:**
- ダッシュボード, 商品管理, 注文管理, 顧客管理, 設定
- 折りたたみ可能（アイコンのみモード）
- 現在のページをハイライト
- モバイル時はハンバーガーメニュー

**ヘッダー:**
- パンくずリスト
- 通知ベル（未読バッジ付き）
- ユーザーアバター → ドロップダウン（プロフィール, ログアウト）

**共通UI:**
- トースト通知（成功/エラー/警告）
- 確認モーダル（削除などの破壊的操作時）
- ローディングスケルトン（データ取得中）
- 空状態（EmptyState）の表示
- エラーバウンダリ

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
## 4. 権限管理（RBAC）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

以下の3ロールを実装:

| 権限 | Admin | Manager | Staff |
|------|-------|---------|-------|
| ダッシュボード閲覧 | ✅ | ✅ | ✅ |
| 商品の作成・編集 | ✅ | ✅ | ✅ |
| 商品の削除 | ✅ | ✅ | ❌ |
| 注文ステータス変更 | ✅ | ✅ | ✅ |
| 返金処理 | ✅ | ✅ | ❌ |
| 顧客情報の編集 | ✅ | ✅ | ❌ |
| CSV出力 | ✅ | ✅ | ❌ |
| ユーザー管理・ロール変更 | ✅ | ❌ | ❌ |
| システム設定 | ✅ | ❌ | ❌ |

実装方針:
- Hono API層での権限チェック必須
- Supabase RLS (Row Level Security) を活用
- フロントでは権限に応じてUI要素の表示/非表示を切り替え
- 権限不足時は403ページまたはトースト通知

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
## 5. データモデル
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

以下のエンティティをSupabase (PostgreSQL) に設計してください:
- User（id, name, email, role, avatar_url, created_at, updated_at）
- Product（id, name, description, sku, category_id, price, sale_price, sale_start, sale_end, stock, status, images[], tags[], slug, meta_title, meta_description, deleted_at, created_at, updated_at）
- Category（id, name, parent_id, created_at）
- Order（id, order_number, customer_id, status, payment_status, payment_method, subtotal, shipping, discount, tax, total, tracking_number, notes, created_at, updated_at）
- OrderItem（id, order_id, product_id, product_name, quantity, unit_price, subtotal）
- Customer（id, name, email, phone, notes, created_at, updated_at）
- Address（id, customer_id, postal_code, prefecture, city, line1, line2, is_default）

※ Supabase の命名規則に合わせて snake_case を使用。
※ RLS ポリシーも合わせて設計してください。

## 6. 非機能要件
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- **レスポンシブ**: デスクトップ優先、タブレットでも操作可能に
- **ダークモード**: ライト/ダークの切り替え対応（Tailwind CSS dark variant使用）
- **国際化対応**: 日本語をデフォルト、将来的に英語追加を想定した構成
- **パフォーマンス**: テーブルのページネーション、Supabaseクエリの最適化
- **アクセシビリティ**: キーボードナビゲーション、適切なaria属性（shadcn/ui が標準対応）
- **エラーハンドリング**: API失敗時のリトライUI、ユーザーへの日本語エラーメッセージ
- **Cloudflare制約**: Workers の10MBサイズ制限を意識、Node.js APIに依存しない

## 8. コード品質ルール
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- TypeScriptのstrictモード、any禁止
- コンポーネントは1ファイル150行以内を目安に分割
- カスタムフックでロジックを分離（useProducts, useOrders等）
- Supabase呼び出しは lib/ 層のサービス関数に集約（画面から直接書かない）
- Hono APIルートは functions/ に集約
- 定数・型定義は専用ファイルに分離
- 日本語コメントで要所にコメントを記載
- ファイル・ディレクトリ構成は最初に提示し、承認を得てから実装開始
- 環境変数はフロント側 VITE_ プレフィックス、Workers側は wrangler で管理
```
